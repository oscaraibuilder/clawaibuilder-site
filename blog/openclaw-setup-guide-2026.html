Step 1: Provision Your VPS

DigitalOcean, Hetzner, or Hostinger all work. The minimum spec for this openclaw setup guide 2026: 2 vCPU, 2GB RAM, 40GB SSD. Ubuntu 22.04 LTS.

# After SSH into your fresh VPS:
apt update && apt upgrade -y
apt install -y curl wget git ufw fail2ban

# Create a non-root user (good practice)
adduser openclaw
usermod -aG sudo openclaw

# Basic firewall
ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable
Don't skip the firewall. A misconfigured VPS with OpenClaw running is an open door. At minimum: SSH only on key auth, UFW enabled, fail2ban running.

Step 2: Install Node.js

OpenClaw requires Node.js 18 or later. The system package manager often gives you something ancient. Use the official installer:

# Install Node.js 22 LTS via NodeSource
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
apt install -y nodejs

# Verify
node --version   # should show v22.x
npm --version

Step 3: Install OpenClaw

The openclaw setup guide 2026 version. OpenClaw is installed globally via npm:

npm install -g openclaw

# Verify installation
openclaw --version

# Create your workspace directory
mkdir -p ~/workspace
cd ~/workspace

# Initialise OpenClaw
openclaw init

The init command walks you through creating your SOUL.md, USER.md, and basic workspace structure. Don't skip this — these files are how your agent knows who it is and who it's serving.

Step 4: Configure Your API Keys

OpenClaw needs at minimum one LLM provider. Anthropic's Claude is the default and best-supported:

# Set environment variables (add to ~/.bashrc or /etc/environment)
export ANTHROPIC_API_KEY="sk-ant-api03-your-key-here"

# Optional: OpenAI as fallback
export OPENAI_API_KEY="sk-your-key"

# Apply
source ~/.bashrc
Get your Anthropic key at console.anthropic.com. Start with a $20 credit to test. Typical daily usage for a personal agent runs $0.50–2.00/day depending on how active you make it.

Step 5: Set Up the Gateway

The gateway is what makes OpenClaw accessible remotely — from your phone, Telegram, anywhere. This is the most common point of failure in openclaw setup, and the part most guides gloss over.

# Start the gateway
openclaw gateway start

# Check status
openclaw gateway status

# Should show: Gateway running on port 3000

If you want the gateway accessible on a proper domain (recommended), you need a reverse proxy. Nginx is standard:

apt install -y nginx certbot python3-certbot-nginx

# Create nginx config
cat > /etc/nginx/sites-available/openclaw << 'EOF'
server {
    listen 80;
    server_name yourdomain.com;
    
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}
EOF

ln -s /etc/nginx/sites-available/openclaw /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx

# Get SSL cert
certbot --nginx -d yourdomain.com

Step 6: Connect Telegram

Telegram is the recommended channel in this openclaw setup guide 2026. It's reliable, works everywhere, and has excellent bot support.

Create your bot

  1. Open Telegram, search for @BotFather
  2. Send /newbot
  3. Follow the prompts — pick a name and username
  4. Copy the token (looks like 1234567890:ABCdef...)
# Add to your environment
export TELEGRAM_BOT_TOKEN="1234567890:ABCdef..."

# Configure in OpenClaw
openclaw config set channels.telegram.token "$TELEGRAM_BOT_TOKEN"
openclaw config set channels.telegram.enabled true

Set the webhook

curl -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/setWebhook" \
  -d "url=https://yourdomain.com/telegram/webhook"
Common gotcha: If your gateway isn't on HTTPS, the Telegram webhook will silently fail. You need a valid SSL cert. Self-signed doesn't work. Use certbot as above.

Step 7: Set Up Memory

OpenClaw's memory system is what makes it actually useful long-term. Without it, every session starts from scratch.

# In your workspace, create the memory structure
mkdir -p ~/workspace/memory

# Create today's daily note
DATE=$(date +%Y-%m-%d)
touch ~/workspace/memory/${DATE}.md

# Create MEMORY.md for long-term context
cat > ~/workspace/MEMORY.md << 'EOF'
# Long-Term Memory

## About Me
[Fill in who you are, what you're building, key context]

## Projects
[Current projects and status]

## Preferences
[How you like things done]
EOF

Your agent will read these files at the start of each session. The more context you give it, the better it performs. This is the single most-overlooked part of any openclaw setup guide.

Step 8: Run OpenClaw as a Service

For 24/7 operation, you need OpenClaw to start automatically and restart if it crashes:

cat > /etc/systemd/system/openclaw.service << 'EOF'
[Unit]
Description=OpenClaw AI Agent
After=network.target

[Service]
Type=simple
User=openclaw
WorkingDirectory=/home/openclaw/workspace
ExecStart=/usr/bin/openclaw start
Restart=always
RestartSec=10
EnvironmentFile=/home/openclaw/.env

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable --now openclaw
systemctl status openclaw
# Create .env file with your API keys
cat > /home/openclaw/.env << 'EOF'
ANTHROPIC_API_KEY=sk-ant-api03-your-key
TELEGRAM_BOT_TOKEN=1234567890:ABCdef
NODE_ENV=production
EOF
chmod 600 /home/openclaw/.env

What Actually Goes Wrong

After doing this openclaw setup guide process many times, these are the real failure modes:

1. Telegram webhook 404

Your nginx config is wrong, or the gateway isn't running. Check openclaw gateway status and systemctl status nginx.

2. API key errors at runtime

Environment variables set in ~/.bashrc don't load for systemd services. Use the EnvironmentFile directive as shown above.

3. Memory not persisting

The agent can only read files it knows about. Make sure your SOUL.md references the memory directory path correctly.

4. High API costs

Context bloat. Workspace files that shouldn't be loaded are getting included in every message. Audit your AGENTS.md file list and trim aggressively.

5. Skills not activating

The skill matching is based on description keywords. If your agent isn't picking up a skill, make the <description> in SKILL.md more specific and action-oriented.
